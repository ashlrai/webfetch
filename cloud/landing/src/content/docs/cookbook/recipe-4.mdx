You care about a topic — a news event, a product launch, a public figure — and want to know whenever new CC-licensed imagery lands. `webfetch watch` handles the diffing and persistence; you handle what to do with new hits.

> **Try it:** a free API key at [app.getwebfetch.com/signup](https://app.getwebfetch.com/signup) keeps `watch` running without you rotating provider keys.

## Run it

```bash
webfetch watch "ukraine frontline 2026" \
  --interval 6h \
  --license safe-only \
  --providers wikimedia,openverse,flickr \
  --webhook https://hooks.example.com/webfetch
```

On every tick:

1. Runs the search with your flags.
2. Compares to the previous tick's state file at `~/.webfetch/watch/<sha256(query)>.json` (keyed by URL).
3. Emits only the net-new candidates to stdout and to your webhook (POST, JSON body) when configured.

## State file

```json
{
  "query": "ukraine frontline 2026",
  "firstSeen": "2026-04-13T10:00:00Z",
  "lastSeen": "2026-04-13T16:00:00Z",
  "urls": ["https://example.com/a.jpg", "https://example.com/b.jpg"]
}
```

Delete the state file if you want to re-emit everything.

## Running under cron / systemd

Cron every 6h:

```cron
0 */6 * * * webfetch watch "ukraine frontline 2026" --once --webhook https://... >> /var/log/webfetch-watch.log 2>&1
```

`--once` means "one tick, exit". That way cron owns the scheduling, not the long-running webfetch process.

systemd timer:

```ini
# /etc/systemd/system/webfetch-watch@.service
[Service]
Type=oneshot
ExecStart=/usr/local/bin/webfetch watch "%i" --once --webhook %E{WEBHOOK_URL}
```

```ini
# /etc/systemd/system/webfetch-watch@.timer
[Timer]
OnCalendar=*-*-* */6:00:00
Unit=webfetch-watch@%i.service
```

Then `systemctl enable --now webfetch-watch@ukraine-frontline.timer`.

## Webhook payload

```json
{
  "query": "ukraine frontline 2026",
  "tickAt": "2026-04-13T16:00:00Z",
  "new": [
    {
      "url": "...",
      "license": "CC_BY",
      "confidence": 0.95,
      "provider": "wikimedia",
      "attributionLine": "..."
    }
  ]
}
```

## Tips

- Use distinct query strings to give related watchers separate state files.
- If your webhook endpoint isn't idempotent, add request deduplication keyed on the candidate `sha256`.
- For high-volume watchers (thousands of new items per tick), pipe the JSON output to your message queue instead of relying on the synchronous webhook.
